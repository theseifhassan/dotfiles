# Spike Issue Template

Use this template when the work is classified as `spike`. Spikes are time-boxed research — the deliverable is knowledge and a recommendation, not production code.

```markdown
## Research Question
What are we trying to learn or figure out. Frame as a clear, answerable question.

## Time Box
Maximum time to spend: [duration, e.g. "1 day", "half a cycle"]
Stop when the time box expires, even if the answer is incomplete — document what you learned.

## Background
What we already know. Prior art, existing solutions, relevant constraints.
Links to docs, discussions, or prior spikes on related topics.

## Exploration Areas
1. [Area to investigate — what to look into and why]
2. [Area to investigate]
3. [Area to investigate]

## Success Criteria
What constitutes a useful answer. When do we know enough to stop.
It's OK if the answer is "don't do this" — that's a valid outcome.

## Deliverable
What artifact is produced (check all that apply):
- [ ] Decision document with recommendation
- [ ] Proof of concept (throwaway code, not production)
- [ ] Technical analysis with trade-offs
- [ ] Architecture proposal

## Decision Points
What decisions will this spike inform.
What follow-up work will be created based on the outcome.

## Acceptance Criteria
- [ ] Research question answered (or documented as unanswerable within time box)
- [ ] Deliverable produced and shared
- [ ] Recommendation documented with rationale
- [ ] Follow-up issues created if applicable
```
