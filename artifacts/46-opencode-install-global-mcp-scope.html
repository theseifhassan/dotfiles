01

Problem & outcome

The dotfiles currently install Claude Code but do not install OpenCode or configure OpenCode MCP access. The requested outcome is a repo-owned Ansible implementation that installs OpenCode in the server and thin client profiles and configures Linear and Notion MCP servers in OpenCode's global user config.

The original work item also asked for personal/work OpenCode Zen account switching. During scope, that branch was intentionally deferred. This pass should leave provider authentication to OpenCode's native /connect flow and should not introduce Vault-managed Zen keys, wrapper scripts, or per-project account switching behavior.

02

Context

The repo has separate playbook-server.yml and playbook-thin.yml profiles. Both already include the claude role, and the claude role demonstrates the local pattern for installing an AI coding CLI with a curl installer and deploying shell config under XDG-managed zsh configuration.

  • —playbook-server.yml runs the full developer toolchain and includes claude; it should also include opencode.
  • —playbook-thin.yml intentionally omits most developer tooling but includes Claude Code; OpenCode should follow that same AI-assistant exception.
  • —OpenCode docs confirm global config lives at ~/.config/opencode/opencode.json, provider auth lives separately in ~/.local/share/opencode/auth.json, and remote MCP servers can be configured under the mcp key.
03

Decision tree

One entry per resolved branch. Each cites the source that resolved it.

Install surface
Create a dedicated opencode role and include it from both playbook-server.yml and playbook-thin.yml with tag opencode. Resolved by work item text and existing profile structure.
Installer
Use the official installer requested by the work item: curl -fsSL https://opencode.ai/install | bash, passing --no-modify-path so Ansible, not the installer, owns shell config.
MCP scope
Configure Linear and Notion as global user-scope OpenCode MCP servers in ~/.config/opencode/opencode.json, with enabled: true for both.
Auth
Leave OpenCode provider auth manual through /connect; remote MCP OAuth is handled by OpenCode via opencode mcp auth linear and opencode mcp auth notion or first-use prompts.
Deferred branch
Do not implement personal/work Zen account switching in this pass. Capture it as a follow-up, noting that OpenCode supports provider.opencode.options.apiKey and {file:...} substitution for a future Vault-backed solution.
04

Alternatives considered

Why the chosen approach beat what was on the table.

Homebrew install
OpenCode supports Homebrew via anomalyco/tap/opencode.
Rejected because the work item explicitly requests the curl installer, and the existing Claude role uses a curl-install pattern.
Local MCP bridge
Use npx -y mcp-remote for Linear and Notion.
Rejected because OpenCode supports remote MCP servers directly, and both vendors document Streamable HTTP endpoints.
Vault Zen switching
Store personal/work Zen API keys in Ansible Vault and select by project.
Deferred by user direction. Viable later, but not part of this implementation pass.

Constraints & dependencies

  • —Installer constraint: use curl -fsSL https://opencode.ai/install | bash as requested by the work item.
  • —XDG/zsh constraint: shell integration must go through ~/.config/zsh/conf.d/ because this repo owns zsh via XDG paths.
  • —Secret constraint: no API keys or provider credentials should be added while account switching is deferred.
  • —Profile constraint: the role must work in both profiles even though the thin profile omits most developer tooling.
